# 06. 객체 지도

## 기능 설계 vs 구조 설계

### 좋은 설계란

- 좋은 설계는 변경의 여지를 남겨놓는 것
- 안정적인 구조에 기능을 종속시켜야 변경에 유연하게 대처하는 소프트웨어를 만들 수 있음

### 구조와 기능

- 구조
    - 사용자나 이해관계자가 **도메인**에 관해 생각하는 개념과 개념들 간의 관계로 구성
    - **도메인 모델**과 관련이 있음
    - 안정적인 재료
- 기능
    - 사용자의 목표 달성을 위해 시스템이 **책임을 수행**하는 행위
    - **유스케이스 모델**과 관련이 있음
    - 불안정적인 재료

### 도메인 모델

- 개념
    - 도메인 : 사용자가 프로그램을 사용하는 **대상 분야**
    - 모델 : 어떤 영역의 지식을 **추상화**하고 **단순화**한 것
    - 도메인 모델 : 프로그램을 사용하는 대상 분야에 관련된 지식을 추상화하고 단순화한 것
- 도메인 모델 = 멘탈 모델
    - 사용자의 예상대로 프로그램이 작동해야 함을 의미
- 소프트웨어 객체는 현실 객체를 은유한 것이 아님 → **도메인 모델을 은유**

### 유스케이스 모델

- 특성
    - 유스케이스는 시스템과 사용자의 **상호작용을 표현하는 텍스트** (not 다이어그램)
    - 유스케이스는 **여러 시나리오**의 집합
    - 유스케이스는 단순 feature와는 다름. **이야기**를 통해 독립적으로 보이는 feature들을 하나로 묶음
    - 사용자 인터페이스와 관련된 상세 정보와 내부 구현에 대한 정보를 포함하지 않음
    - 유스케이스는 **객체의 구조와 책임에 대해 어떤 정보도 제공하지 않는다**
        - 유스케이스로부터 도메인 모델을 설계하지 마라
- 책임 주도 설계는 도메인 모델과 유스케이스를 통합하는 것
    - 예제 : 이자 계산 기능
        - 시스템이 “중도 해지 이자액을 계산하라”는 메시지를 받는 거대한 객체라고 가정한다.
        - 도메인 모델로부터 메시지를 처리할 객체를 선택하고, 다른 객체에 보내야 하는 메시지가 있는지 확인한다. 이 과정을 통해 객체 간 협력 관계를 창조한다.
