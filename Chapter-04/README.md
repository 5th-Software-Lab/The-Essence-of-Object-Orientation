# 04. 역할, 책임, 협력

## 1. 협력 (Collaboration)

객체들이 시스템의 기능을 구현하기 위해 수행하는 상호작용입니다.

- **시작:** 한 객체가 다른 객체에게 도움을 요청할 때 시작됩니다.
- **과정:** 요청과 응답이 연쇄적으로 이어지며 하나의 목표를 달성합니다.
- **본질:** 협력은 객체의 존재 이유를 결정하며, 객체의 행동을 정의하는 문맥(Context)이 됩니다.

---

## 2. 책임 (Responsibility)

협력에 참여하기 위해 객체가 수행해야 하는 행동의 목록입니다.

### 💼 책임의 분류

| 분류 | 세부 내용 |
| --- | --- |
| **하는 것 (Doing)** | 객체를 생성하거나 계산을 수행하는 등 스스로 하는 것
다른 객체의 행동을 시작시키거나 활동을 조절하는 것 |
| **아는 것 (Knowing)** | 개인적인 정보나 관련 객체에 대해 아는 것
자신이 유도하거나 계산할 수 있는 것에 대해 아는 것 |
- **책임과 인터페이스:** 책임은 객체가 외부에 제공하는 정보와 서비스의 목록입니다. 이는 곧 **공용 인터페이스**를 구성하며, 자연스럽게 **캡슐화**로 이어집니다.
- **책임과 메시지:** 설계 초반에는 '무엇을 할 것인가'에 집중하고 '어떻게 할 것인가'는 나중에 고민합니다. 하나의 책임이 여러 개의 메시지로 분리될 수 있기 때문입니다.

---

## 3. 역할 (Role)

협력 안에서 책임의 집합을 추상화한 것입니다.

- **재사용성:** 역할을 통해 협력을 추상화하면 다양한 객체가 참여할 수 있어 재사용성이 높아집니다.
    - 예: `판사` 역할 (왕 → 여왕으로 교체 가능)
    - 예: `증인` 역할 (모자 장수 → 앨리스로 교체 가능)
- **대체 가능성 (Substitutability):** 어떤 객체가 역할을 수행하려면, 그 역할이 수신할 수 있는 **동일한 메시지**를 이해해야 합니다.
- **추상화:** 역할은 객체의 추상화이며, `역할(일반화)`과 `객체의 타입(특수화)`은 일반화-특수화 관계를 가집니다.

---

## 4. 객체지향 설계 기법

좋은 협력을 만들기 위한 세 가지 대표적인 방법론입니다.

### 🧱 책임 주도 설계 (Responsibility-Driven Design)

- 시스템의 책임을 파악하고, 그 책임을 수행할 적절한 객체를 찾아 책임을 할당하는 방식입니다.
- **데이터보다 행동**을 먼저 결정합니다.

### 🎨 디자인 패턴 (Design Pattern)

- 반복적으로 발생하는 문제에 대해 전문가들이 뽑아낸 '역할, 책임, 협력'의 템플릿입니다.
- 특정 문제를 해결하기 위해 어떤 객체들이 어떤 책임을 가져야 하는지 미리 정의해 둔 것입니다.

### 🧪 테스트 주도 개발 (Test-Driven Development)

- 테스트를 먼저 작성함으로써 객체의 **역할과 책임이 협력 안에서 적절한지** 검증하며 설계하는 방식입니다.
- 단순히 코드를 검증하는 것을 넘어, 객체의 인터페이스를 설계하는 도구로 활용됩니다.
